title: Secrets management
description: Securely store and manage Cloudflare API tokens in your operating system's keyring

The npmctl CLI can securely store your Cloudflare API token in your operating system's keyring, eliminating the need to pass it via command-line flags or environment variables for every command.

## Why use the keyring?

Storing secrets in the OS keyring provides several security benefits:

- **No plaintext files**: Tokens aren't stored in shell history, config files, or environment variable files
- **OS-level encryption**: Your operating system encrypts and manages access to keyring data
- **Convenient**: Set once, use everywhere without passing `--cloudflare-api-token` flags
- **Audit trail**: OS keyrings often provide access logging and security monitoring

## Supported keyring backends

npmctl uses the Python `keyring` library, which supports:

- **macOS**: Keychain
- **Windows**: Credential Manager
- **Linux**: Secret Service (GNOME Keyring), KWallet, or other compatible backends

<Note>
  On Linux, you may need to install and configure a keyring backend. Run `npmctl doctor` to check your system and get installation instructions.
</Note>

## Storing your Cloudflare API token

<Steps>
  <Step title="Get your Cloudflare API token">
    Create an API token in your Cloudflare dashboard:
    1. Go to **My Profile** → **API Tokens**
    2. Click **Create Token**
    3. Use the **Edit zone DNS** template or create a custom token
    4. Required permissions:
       - Zone / DNS / Edit
       - Zone / Zone / Read
    5. Scope to specific zones (domains) as needed
    6. Copy the generated token
  </Step>

  <Step title="Run the cf-token-set command">
    ```bash
    npmctl cf-token-set
    ```
    
    You'll be prompted to enter the token twice (for confirmation). Input is hidden for security.
  </Step>

  <Step title="Verify the token is stored">
    ```bash
    npmctl cf-token-status
    ```
    
    You should see:
    ```text
    Cloudflare token is stored in keyring.
    ```
  </Step>
</Steps>

### Alternative: pass token directly

You can also provide the token directly (not recommended - it may be visible in process lists):

```bash
npmctl cf-token-set --token your_cloudflare_api_token_here
```

<Warning>
  Using `--token` on the command line may expose the token in shell history and process listings. The interactive prompt is more secure.
</Warning>

## Checking token status

To verify whether a Cloudflare API token is stored in the keyring:

```bash
npmctl cf-token-status
```

Possible outputs:

<Accordion title="Token is stored">
  ```text
  Cloudflare token is stored in keyring.
  ```
  
  You can use certificate commands without specifying `--cloudflare-api-token`.
</Accordion>

<Accordion title="No token stored">
  ```text
  No Cloudflare token stored in keyring.
  ```
  
  Run `cf-token-set` to store a token, or provide it via `--cloudflare-api-token` flag or `CLOUDFLARE_API_TOKEN` environment variable.
</Accordion>

<Note>
  The `cf-token-status` command only checks if a token exists - it doesn't display the actual token value or validate it with Cloudflare.
</Note>

## Deleting the stored token

To remove your Cloudflare API token from the keyring:

```bash
npmctl cf-token-delete
```

You'll see a confirmation:
```text
Cloudflare token deleted from OS keyring.
```

After deletion, you'll need to provide the token via `--cloudflare-api-token` flag or `CLOUDFLARE_API_TOKEN` environment variable, or run `cf-token-set` again.

## Token resolution priority

When you run certificate commands, npmctl looks for the Cloudflare API token in this order:

1. **`--cloudflare-api-token` flag** (highest priority)
   ```bash
   npmctl add-cert-cloudflare \
     --domain example.com \
     --cloudflare-api-token your_token_here
   ```

2. **`CLOUDFLARE_API_TOKEN` environment variable**
   ```bash
   export CLOUDFLARE_API_TOKEN=your_token_here
   npmctl add-cert-cloudflare --domain example.com
   ```

3. **OS keyring** (set via `cf-token-set`)
   ```bash
   npmctl add-cert-cloudflare --domain example.com
   ```

If none of these provide a token, the command will fail with an error.

<Tip>
  For interactive use, the keyring is recommended. For CI/CD pipelines and scripts, use environment variables.
</Tip>

## Using stored tokens with certificate commands

Once you've stored a token with `cf-token-set`, you can omit the `--cloudflare-api-token` flag:

### Create a certificate

```bash
npmctl add-cert-cloudflare \
  --domain example.com \
  --domain '*.example.com'
```

npmctl automatically retrieves the token from the keyring.

### Create certificate + proxy host

```bash
npmctl add-proxy-with-cert \
  --domain app.example.com \
  --forward-host 192.168.1.10 \
  --forward-port 8080
```

Both the certificate creation and proxy host setup use the stored token.

## Troubleshooting

<AccordionGroup>
  <Accordion title="Keyring backend not available">
    If you see errors like:
    ```text
    Failed to read secret from keyring: No recommended backend was available
    ```
    
    Run the diagnostics tool:
    ```bash
    npmctl doctor
    ```
    
    This will check your keyring backend and provide distro-specific installation instructions. See the [troubleshooting guide](/guides/troubleshooting) for details.
  </Accordion>
  
  <Accordion title="Token exists but commands still fail">
    The token may be invalid or have insufficient permissions. Try:
    
    1. Delete the stored token:
       ```bash
       npmctl cf-token-delete
       ```
    
    2. Verify token permissions in Cloudflare dashboard
    
    3. Generate a new token with correct permissions:
       - Zone / DNS / Edit
       - Zone / Zone / Read
    
    4. Store the new token:
       ```bash
       npmctl cf-token-set
       ```
  </Accordion>
  
  <Accordion title="Different token for different domains">
    npmctl currently stores only one Cloudflare API token in the keyring. If you need different tokens for different domains:
    
    **Option 1**: Create a Cloudflare API token with access to all required zones
    
    **Option 2**: Use environment variables or flags for specific commands:
    ```bash
    CLOUDFLARE_API_TOKEN=token_for_domain1 npmctl add-cert-cloudflare --domain domain1.com
    CLOUDFLARE_API_TOKEN=token_for_domain2 npmctl add-cert-cloudflare --domain domain2.com
    ```
  </Accordion>
  
  <Accordion title="Permission denied errors on Linux">
    If keyring operations fail with permission errors:
    
    1. Ensure you're running from a normal user session (not a minimal headless shell)
    2. Check that D-Bus is running:
       ```bash
       echo $DBUS_SESSION_BUS_ADDRESS
       ```
    3. If empty, you may need to install `dbus-user-session`:
       ```bash
       sudo apt install dbus-user-session  # Ubuntu/Debian
       ```
  </Accordion>
</AccordionGroup>

## Security best practices

<CardGroup cols={2}>
  <Card title="Rotate tokens regularly" icon="rotate">
    Periodically create new Cloudflare API tokens and update the stored token:
    ```bash
    npmctl cf-token-delete
    npmctl cf-token-set
    ```
  </Card>
  
  <Card title="Use minimal permissions" icon="shield">
    Create Cloudflare API tokens with only the permissions needed:
    - Zone / DNS / Edit
    - Zone / Zone / Read
    
    Scope to specific zones rather than all zones.
  </Card>
  
  <Card title="Use keyring for interactive use" icon="desktop">
    Store tokens in the keyring for day-to-day CLI use.
  </Card>
  
  <Card title="Use env vars for automation" icon="robot">
    Use `CLOUDFLARE_API_TOKEN` environment variable in CI/CD pipelines and scripts instead of the keyring.
  </Card>
</CardGroup>
